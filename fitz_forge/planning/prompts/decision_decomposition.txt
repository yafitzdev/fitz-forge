## Decision ordering rules

1. Start with PATTERN decisions: "What is the right architectural approach?" comes before design details.
2. Then INTERFACE decisions: "What method signature changes are needed on X?" depends on the pattern choice.
3. Then INTEGRATION decisions: "How does component A connect to component B?" depends on interface decisions.
4. Then SCOPE decisions: "What is out of scope for v1?" can reference earlier decisions.
5. Finally TECHNICAL decisions: implementation details depend on all above.

## Pattern decisions must evaluate build-new vs extend-existing — when applicable

The build-new-vs-extend question applies when the task introduces a **new capability concept** (sharing, explainability, auth, billing) that could plausibly live in its own module with its own data model. Without this explicit meta-question, the plan will anchor onto whatever existing class happens to be semantically nearest and miss cleaner architectural separations.

**SKIP this question and proceed directly to interface decisions when the task is clearly adding new BEHAVIOR to an existing class** (streaming on an engine, caching on a repository, metrics on a service, pagination on a query). In those cases the capability is a method/feature on a known surface — proposing a parallel module would be over-engineering and produces worse plans.

How to tell which camp a task is in:
- Could the task's capability name plausibly be a module name (e.g. "CollectionShare", "RankingExplainer", "AuthProvider")? → **new-capability concept**, ASK the build-new question as d1.
- Is the task a new method on a specific known class (e.g. "stream_answer on Engine", "cache_lookup on Repository")? → **behavior-on-existing**, SKIP the build-new question; start with interface decisions on that class.

Use the user prompt to decide: if it names a specific existing class/file as the thing to modify, it is extension territory. If it names a new capability without pinning an owner, it is new-capability territory.

A pattern decision IS ALLOWED to propose candidate new files/modules that do not yet exist in the manifest (when the new-capability case applies). Subsequent design decisions then name those new paths directly. Resolution still reads 1-3 files from the manifest to evaluate the tradeoff (typically the nearest existing mechanism).

Good pattern decisions look like:
- (new-capability case) "Should collection sharing live in a new `collection-share` module with its own Prisma model, or as new fields on the existing `Shortcode` model in `prisma/schema.prisma`?"
- (new-capability case) "Should ranking explanations be added as a new `RankingExplainer` service, or as `metadata` enrichment on the existing `Address` / `Ranker` pipeline?"
- (behavior-on-existing case) "What is the signature of the new `stream_answer` method on `Engine` in `engine.py`, and where does it hook into the existing pipeline?"

Bad pattern decisions look like:
- "How should the relationship between `Shortcode` and `Collection` be modeled in `prisma/schema.prisma`?" (pre-commits to extending Shortcode for a new-capability task)
- "What is the method signature for a new `createCollectionShortcode` method in `shortcode.service.ts`?" (interface decision disguised as pattern — the mechanism is already chosen)
- "Should streaming live in a new `StreamingEngine` module or extend existing `Engine`?" (over-asking for a behavior-on-existing task where the user prompt already names `Engine` as the host)

Only once the mechanism is settled (either via the build-new decision OR directly because the host class is obvious) can you ask interface and integration questions.

## Downstream decisions must not pre-commit to an upstream branch

Every decision that `depends_on` a pattern decision must reference the upstream resolution abstractly — do NOT bake a specific mechanism into the question. The resolver answers each decision independently from its local phrasing; if d2 says "If extending X, what fields...", it will answer the extension branch even when d1 chose the alternative. The plan then ends up as a Frankenstein of mixed mechanisms.

- BAD d2 (given d1 is build-new-vs-extend): "If extending the existing `Shortcode` model, what new fields are required?"
- GOOD d2: "Given the mechanism chosen in d1, what Prisma schema changes are required to support the new capability?"
- BAD d3: "What is the method signature for `createCollectionShortcode` in `shortcode.service.ts`?" (pre-commits to Shortcode service)
- GOOD d3: "Given d1's chosen host service, what is the create/revoke method signature for the new capability?"

Phrase downstream questions so they are answerable *whichever* way the pattern decision goes. Keep the `relevant_files` list to the candidate files on both sides of the pattern fork (typically the existing mechanism's files), so the resolver can still ground its answer in real source.

## Chain completeness rule

When the task adds new behavior at an entry point (API route, CLI command, SDK method), trace the FULL call chain from the entry point to the implementation layer using the call graph edges. Generate a decision for EACH file in the chain. Skipping a middle layer (e.g., a service wrapper between a route and an engine) creates an impossible implementation gap — the entry point can't call methods that don't exist on its direct dependency.
